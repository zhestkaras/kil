# **ТЕХНИЧЕСКОЕ ЗАДАНИЕ НА СОЗДАНИЕ АВТОМАТИЗИРОВАННОЙ СИСТЕМЫ**

## **1. ОБЩИЕ СВЕДЕНИЯ**

### **1.1. Полное наименование и условное обозначение:**
- Полное наименование: Автоматизированная система управления учебным процессом "TaskFlow"
- Условное обозначение: АСУП "TaskFlow"

### **1.2. Шифр темы:** TF-2024-01

### **1.3. Организации-участники:**
- **Заказчик:** Государственное профессиональное образовательное учреждение "Колледж информационных технологий"
- **Разработчик:** Команда разработчиков под руководством [ФИО руководителя проекта]

### **1.4. Основание для создания:**
1. Приказ директора колледжа № 145 от 15.09.2024 "О внедрении цифровых технологий в учебный процесс"
2. План развития информатизации колледжа на 2024-2026 гг.
3. Технико-экономическое обоснование "Внедрение системы управления учебными задачами"

### **1.5. Плановые сроки работ:**

- Окончание работ: 01.04.2025
- Срок эксплуатации: не менее 5 лет

### **1.6. Финансирование:**

- Порядок финансирования: Поэтапно согласно календарному плану

## **2. ЦЕЛИ И НАЗНАЧЕНИЕ СОЗДАНИЯ АВТОМАТИЗИРОВАННОЙ СИСТЕМЫ**

### **2.1. Цели создания АС:**
1. **Технические цели:**
   - Сокращение времени на управление учебными задачами на 70%
   - Увеличение своевременности сдачи заданий с 65% до 92%
   - Снижение количества пропущенных дедлайнов с 3.2 до 0.5 на студента в месяц
   - Уменьшение административной нагрузки преподавателей на 75%

2. **Образовательные цели:**
   - Повышение средней успеваемости на 0.5 балла
   - Увеличение процента качественной успеваемости с 45% до 65%
   - Снижение уровня стресса студентов с 8.2 до 4.5 баллов по 10-балльной шкале

3. **Организационные цели:**
   - Автоматизация 85% рутинных операций по учету учебных задач
   - Снижение времени на формирование отчетности на 87%
   - Внедрение проактивной системы контроля учебного процесса

### **2.2. Назначение АС:**
- **Вид автоматизируемой деятельности:** Управление учебным процессом, контроль выполнения заданий, мониторинг успеваемости
- **Область применения:** Учебные группы колледжа (первый этап - 5 групп, далее масштабирование)
- **Основные пользователи:**
  - Студенты (250 человек на первом этапе)
  - Преподаватели (25 человек)
  - Кураторы учебных групп (10 человек)
  - Администрация колледжа

## **3. ХАРАКТЕРИСТИКА ОБЪЕКТА АВТОМАТИЗАЦИИ**

### **3.1. Основные сведения об объекте:**
- **Объект автоматизации:** Учебный процесс колледжа информационных технологий
- **Количество студентов:** 1200 человек
- **Количество групп:** 48 учебных групп
- **Количество преподавателей:** 85 человек
- **Специальности:** 8 направлений подготовки
- **Ежегодный объем заданий:** ~15 000 заданий различных типов

### **3.2. Текущее состояние:**
1. **Процессы, подлежащие автоматизации:**
   - Выдача и прием учебных заданий
   - Контроль сроков выполнения
   - Учет успеваемости студентов
   - Формирование отчетности
   - Коммуникация между участниками учебного процесса

2. **Существующие проблемы:**
   - Разрозненность информации (тетради, электронная почта, мессенджеры)
   - Ручной учет дедлайнов и выполнений
   - Отсутствие единой системы контроля нагрузки
   - Высокие временные затраты на администрирование
   - Сложности в получении оперативной статистики

### **3.3. Условия эксплуатации:**
- **Климатические условия:** Умеренный климат, температура +15°C...+25°C
- **Электропитание:** 220В ±10%, 50Гц
- **Помещения:** Учебные аудитории, компьютерные классы, административные кабинеты
- **Сетевые условия:** Локальная сеть колледжа, скорость доступа к интернету не менее 100 Мбит/с

   - Регистрация и аутентификация

   - Профили пользователей

2. **Подсистема управления заданиями:**

   - Назначение сроков выполнения
   - Контроль выполнения

3. **Подсистема мониторинга и аналитики:**
   - Сбор статистики успеваемости

4. **Подсистема коммуникации:**
   - Внутренняя система уведомлений
   - Комментарии и обратная связь
   - Уведомления о событиях

5. **Подсистема интеграции:**
   - API для внешних систем
   - Импорт/экспорт данных
   - Синхронизация с существующими системами

#### **4.1.3. Требования к информационному взаимодействию:**
- **Протоколы:** HTTP/HTTPS, REST API
- **Форматы данных:** JSON, XML
- **Стандарты обмена:** OpenAPI 3.0 для документации API

#### **4.1.4. Интеграционные требования:**
- **Смежные системы:**

#### **4.1.5. Режимы функционирования:**

- **Резервный режим:** Переход на упрощенную версию при сбоях

#### **4.1.6. Требования по диагностированию:**

- Система логирования и аудита действий

#### **4.1.7. Перспективы развития:**
